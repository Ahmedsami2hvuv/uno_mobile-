# خطة المشروع الكاملة — بدون Expo Go، مستودع جديد، ريلوي، APK، وتحديثات مباشرة

## رأيي بالخطتك

فكرتك مناسبة وواضحة. هذا ترتيب تنفيذها مع توضيح نقطة "المستخدم يشوف التحديثات مباشرة".

---

## ١) نترك تطبيق Expo Go ونبدأ شغل جديد

- **Expo Go** = للتجربة أثناء التطوير فقط.
- النسخة النهائية = **APK** يثبّته المستخدم (من رابط تحميل أو من المتجر لاحقاً). المستخدم **لا يحتاج** Expo Go.
- المشروع الحالي (uno-mobile) يبقى هو أساس التطبيق؛ ننظّفه ونجهّزه كمستودع تطبيق واحد.

---

## ٢) نأخذ الفكرة والمنطق والأزرار من مستودع أونو الأصلي

- المستودع الأصلي: [Ahmedsami2hvuv/uno](https://github.com/Ahmedsami2hvuv/uno) (البوت).
- نأخذ منه:
  - **النصوص والقوائم** (من i18n) — تم بالفعل في `src/strings.js`.
  - **فكرة القائمة والأزرار** — تم في الشاشة الرئيسية.
  - **قواعد اللعب** — منطق اللعبة في `src/logic/unoLogic.js` مأخوذ من فكرة البوت.
- المستودع الجديد للتطبيق **لا ينسخ** كود البوت؛ يبقى البوت على حاله. نحن نستلهم الفكرة والمنطق فقط.

---

## ٣) مستودع جديد للتطبيق (مسوي الآن)

- تنشئ مستودعاً على GitHub للتطبيق فقط (مثلاً **uno-app** أو **uno-mobile**).
- تضع فيه:
  - كل ملفات التطبيق (شاشات، منطق، قوانين، حاسبة، ربط ريلوي).
  - ملف **`src/railwayConfig.js`** — فيه رابط ريلوي فقط (ربط التطبيق بريلوي).
  - **لا** تضع فيه ملفات البوت؛ البوت يبقى في مستودعه.

---

## ٤) ربط المستودع بريلوي

- ريلوي **لا يشغّل تطبيق الموبايل**. ريلوي يشغّل **سيرفر (API)** فقط.
- لذلك:
  - **مستودع ١ (التطبيق):** فيه كود الموبايل. **لا** نربطه بريلوي للـ "تشغيل". نربطه بريلوي فقط لو عندك CI يبني الـ APK (اختياري).
  - **مستودع ٢ (الـ API):** فيه api_server.py و database.py و requirements.txt و Procfile. **هذا** نربطه بريلوي — ريلوي يسحب منه ويشغّل السيرفر.
- التطبيق (الـ APK) يتصل **بريلوي** عبر الرابط في `railwayConfig.js`. إذن: **التطبيق مرتبط بريلوي**، و**ريلوي مرتبط بمستودع الـ API**.

---

## ٥) ملف APK (تطبيق) مرتبط بريلوي

- نبني **APK** من مستودع التطبيق عبر **EAS Build** (Expo):
  ```bash
  npx eas build --platform android --profile preview
  ```
- الـ APK يحتوي بداخله **رابط ريلوي** من `railwayConfig.js`. المستخدم يثبّت الـ APK ويفتح التطبيق → التطبيق يرسل الطلبات إلى ريلوي. = **تطبيق APK مرتبط بريلوي** ✅

---

## ٦) نعدّل على ملفات GitHub → المستخدم يشوف التحديثات مباشرة

هنا نقطة مهمة:

- لو عدّلت على GitHub وبنيت **APK جديد** فقط، المستخدم اللي عنده النسخة القديمة **لن** يحصل على التحديث إلا إذا حمّل وثبّت الـ APK الجديد.
- عشان **المستخدم يشوف التحديثات مباشرة** (بدون إعادة تثبيت APK كل مرة) نستخدم **Expo EAS Update**:
  1. تعدّل الملفات على GitHub (أو محلياً ثم تدفع).
  2. تشغّل: `npx eas update --branch production` (أو branch تختاره).
  3. المستخدم يفتح التطبيق (نفس الـ APK القديم) → التطبيق يتحقق من التحديثات ويحمّل النسخة الجديدة (JS والأصول) → **يشوف التحديث مباشرة**.

يعني: **نعم** — نعدّل على ملفات GitHub، ونستخدم **EAS Update** حتى المستخدم يشوف التحديثات مباشرة بدون تحميل APK جديد (ما عدا لو غيّرت نسخة الـ native أو إعدادات البناء).

---

## ترتيب التنفيذ المقترح

| الخطوة | ماذا تفعل |
|--------|-----------|
| 1 | مستودع جديد للتطبيق على GitHub → ترفع فيه كود التطبيق الحالي (بما فيه railwayConfig.js و eas.json). |
| 2 | مستودع الـ API (uno-app-api) مربوط بريلوي — ريلوي ياخذ منه السيرفر. |
| 3 | بناء APK من مستودع التطبيق: `npx eas build --platform android --profile preview` — توزّع الرابط للمستخدمين. |
| 4 | تفعيل التحديثات المباشرة: `npx eas update:configure` ثم عند كل تعديل: عدّل على GitHub ثم `npx eas update --branch production`. |

---

## ملخص

- نترك الاعتماد على Expo Go للمستخدم النهائي؛ نوزّع **APK** مرتبط بريلوي.
- نأخذ الفكرة والمنطق والأزرار من مستودع أونو الأصلي (بدون نسخ مستودع البوت).
- مستودع جديد للتطبيق فقط؛ ريلوي مربوط بمستودع الـ API.
- APK = مرتبط بريلوي عبر **railwayConfig.js**.
- عشان المستخدم يشوف التحديثات مباشرة: نعدّل على ملفات GitHub ونشغّل **EAS Update** — بدون إعادة تثبيت APK.

تم إضافة **eas.json** في المشروع لبناء الـ APK. بعد ربط المشروع بـ Expo (eas init أو من الموقع)، تستخدم `eas update:configure` ثم `eas update` بعد كل تعديل على GitHub حتى يشوف المستخدم التحديثات مباشرة.
