# خارطة الطريق — تطبيق أونو vs بوت أونو

## وضع التطبيق الحالي

التطبيق يعمل **بدون سيرفر** و**بدون قاعدة بيانات** و**بدون حسابات**. ما يعمل الآن:

| الميزة | التطبيق | البوت |
|--------|---------|-------|
| اللعب مع البوت (AI) | ✅ يعمل | ✅ |
| قوانين اللعب | ✅ يعمل | ✅ |
| حاسبة النقاط | ✅ يعمل | ✅ |
| **قاعدة البيانات** | ❌ لا يوجد | ✅ (users, rooms, games, stats) |
| **الحساب / اليوزر** | ❌ لا يوجد | ✅ تسجيل، دخول، بروفايل |
| **اللعب العشوائي** | ❌ لا يوجد | ✅ مطابقة مع لاعب آخر |
| **لعب مع الأصدقاء** | ❌ لا يوجد | ✅ غرف، إنشاء، انضمام بكود |
| **الإحصائيات / لوحة المتصدرين** | ❌ لا يوجد | ✅ |
| **الشارات** | ❌ لا يوجد | ✅ |
| **المساعدة** | ❌ لا يوجد | ✅ طلب مساعدة للإدارة |
| **النشر بالمجتمع** | ❌ لا يوجد | ✅ قناة/مجتمع |
| **تغيير اللغة** | ❌ لا يوجد | ✅ عربي، إنجليزي، فارسي |

---

## لماذا التطبيق لا يساوي البوت؟

البوت يعمل على **سيرفر** (مثلاً Repl.it أو VPS) ويرتبط بـ:

1. **قاعدة بيانات** (PostgreSQL أو غيره) فيها:
   - المستخدمون (user_id, name, username, password, points, language, …)
   - الغرف (room_id, code, host, players, game_state, …)
   - المباريات والإحصائيات والشارات

2. **منطق اللعب الجماعي**: مطابقة عشوائية، غرف، دور كل لاعب، التحدي، إلخ.

3. **Telegram API**: لإرسال الرسائل والزرائر للمستخدمين.

التطبيق الحالي **لا يتصل بأي سيرفر**. كل شيء يعمل على الجهاز (لعب ضد بوت محلي، قوانين، حاسبة). لذلك لا يوجد حسابات ولا لعب عشوائي ولا غرف أصدقاء ولا شارات ولا مساعدة ولا نشر.

---

## ماذا نحتاج ليقترب التطبيق من البوت؟

### 1) سيرفر خلفي (Backend API)

- إما **توسيع البوت** ليفتح واجهة ويب (Web API) يستقبل طلبات من التطبيق.
- أو **سيرفر جديد** (Node.js، Python FastAPI، إلخ) يتكلم مع **نفس قاعدة البيانات** التي يستخدمها البوت (أو نسخة مزامنة منها).

هذا السيرفر يجب أن يقدّم على الأقل:

- **تسجيل / دخول** (إنشاء حساب، تسجيل دخول، جلب بروفايل).
- **لعب عشوائي**: طلب مباراة → وضع المستخدم في صف انتظار → عند وجود لاعب ثاني، بدء مباراة وإرسال الحالة للطرفين.
- **غرف الأصدقاء**: إنشاء غرفة (كود أو رابط)، انضمام بكود، قائمة اللاعبين في الغرفة، بدء اللعب.
- **الإحصائيات**: نقاط، لوحة متصدرين، سجل مباريات.
- **الشارات**: حفظ وعرض شارة كل مستخدم.
- **المساعدة**: إرسال طلب مساعدة للإدارة.
- **النشر بالمجتمع**: ربط مع قناة/مجتمع حسب تصميم البوت.

### 2) قاعدة البيانات

- يجب أن تكون **موحدة** بين البوت والتطبيق (نفس الجداول: users, rooms, games، إلخ) حتى:
  - المستخدم الذي يسجّل من التطبيق يظهر في البوت والعكس.
  - الغرف والمباريات واحدة.

### 3) الربط في التطبيق (Repl.it / السيرفر)

- في التطبيق نحدد **عنوان السيرفر** (مثلاً `https://your-bot.repl.co` أو عنوان VPS).
- كل طلب (تسجيل، دخول، إنشاء غرفة، انضمام، إرسال ورقة، إلخ) يذهب إلى هذا السيرفر.
- اللعب الجماعي يحتاج **اتصال حي** (WebSocket أو long polling) حتى يصل تحديث الدور والورقة النازلة فوراً للطرفين.

### 4) ما يبنى داخل التطبيق بعد وجود الـ API

- شاشة **تسجيل / دخول** (اسم، يوزر، كلمة سر).
- شاشة **حسابي** (بروفايل، نقاط، شارة، إعدادات).
- شاشة **لعب عشوائي** (طلب مباراة، انتظار، ثم شاشة لعب مع بيانات من السيرفر).
- شاشة **لعب مع الأصدقاء** (إنشاء غرفة، انضمام بكود، قائمة غرف، ثم لعب).
- شاشة **الإحصائيات** (لوحة متصدرين، سجل مباريات).
- **الشارات** (عرض في البروفايل + انشر شارتك إن وُجدت في الـ API).
- **المساعدة** (نموذج أو زر يرسل طلب مساعدة إلى السيرفر).
- **النشر بالمجتمع** (حسب ما يوفره السيرفر: رابط نشر، إلخ).
- **تغيير اللغة** (حفظ في الحساب عبر الـ API وعرض النصوص المناسبة).

## خلاصة

- **الآن:** التطبيق = لعب ضد بوت + قوانين + حاسبة، **بدون** قاعدة بيانات أو حسابات أو سيرفر.
- **حتى يصبح قريباً من البوت:** نحتاج **سيرفر (API) + قاعدة بيانات + حسابات**، ثم ربط التطبيق بهذا السيرفر وبناء شاشات: الحساب، اللعب العشوائي، لعب مع الأصدقاء، الإحصائيات، الشارات، المساعدة، النشر بالمجتمع، وتغيير اللغة.

القائمة الرئيسية في التطبيق تعرض كل خيارات البوت؛ الخيارات التي تحتاج سيرفراً تظهر حالياً كـ «قريباً — يحتاج سيرفر وقاعدة بيانات وحسابات» حتى يتم بناء الـ API والربط.

## ربط التطبيق بحسابي وقاعدة البيانات (تم البدء)

- **مجلد `uno-api`** في المشروع يحتوي على **api_server.py** — انسخه إلى مستودع البوت [uno](https://github.com/Ahmedsami2hvuv/uno) وثبّت `fastapi` و `uvicorn` وشغّله على نفس السيرفر (Repl.it/Railway) الذي يشغّل البوت.
- **التطبيق** فيه شاشة **حسابي** وشاشة **تسجيل/دخول** وربط مع الـ API (تسجيل، دخول، بروفايل). غيّر **BASE_URL** في **src/api.js** إلى عنوان سيرفرك بعد تشغيل الـ API.
- بعدها يصبح الحساب **واحداً** بين البوت والتطبيق (نفس جدول `users` وقاعدة البيانات).
